## 2025-03-02 - [PCI Bus Scanner & Thread Safety]
**Anomaly:** The PCI Configuration Access Mechanism (CAM) utilizes a two-step read via CPU ports 0xCF8 (Address) and 0xCFC (Data). Running this inline is subject to race conditions if a hardware interrupt fires between the address setup and the data read.
**Resolution:** Encapsulated the Port access behind a globally static `spin::Mutex<PciPort>`. To completely eliminate deadlocks, the `.lock()` operation is executed exclusively inside an `x86_64::instructions::interrupts::without_interrupts` closure. Successfully discovered the Intel XHCI controller by parsing multi-function headers and matching Class 0x0C, Subclass 0x03, Prog IF 0x30, handling 64-bit BARs safely.
## 2025-03-02 - [XHCI Halt, Reset, & Memory Binding]
**Anomaly:** Directly assuming the heap memory mapped exactly to the raw physical memory linear scale during hardware bindings. `linked_list_allocator` utilizes virtual mappings that have no direct translation without consulting the page table. Subtracting the generic `physical_memory_offset` to derive a physical address creates catastrophic DMA writes into random RAM.
**Resolution:** Allocated a strict 64-byte aligned Command Ring using `alloc::alloc::alloc_zeroed`. Refused to execute raw linear subtraction. Employed `mapper.translate_addr` via a formal Level-4 page table walk to resolve the virtual heap pointer into the exact physical hardware backing frame required for the `CRCR` register. Successfully halted the rogue UEFI XHCI engine using volatile reads/writes on `USBCMD` and `USBSTS`, cleanly initialized the `CRCR`, and woke the Intel USB stack.
