## 2026-03-28 - Bypassing ListView items_changed Optimization for Zero-Copy State Alignment
**Anomaly:** The `left_expand_btn` and `right_expand_btn` in `comms.rs` depressed but failed to toggle the message expansion. **Resolution:** Extracted the truncation and DOM-mutation logic into a pure Can-Am helper function (`apply_expansion_state`). The click handlers in `connect_setup` now use `unsafe { item.data("widgets") }` to retrieve the `BubbleWidgets` struct and invoke the helper directly. This surgically mutates the view state synchronously, bypassing the `ListView`'s redraw cycle entirely without splicing the data store.

## 2026-03-28 - Enforcing Responsive Layout Stagger in Chat View
**Anomaly:** Chat bubbles failed to visually stagger on wide screens, sitting directly over each other. The `msg_label` natural width request (`85 chars`) was consuming nearly all available window space, preventing `halign` from creating a visible left/right flow.

## 2026-03-28 - Severing GTK4 hexpand Propagation on Ultra-Wide Displays
**Anomaly:** On a 2880px wide viewport, user bubbles remained stuck on the left and stretched across the screen instead of staggering right. The `.hexpand(true)` property on `meta_label` was bubbling up the widget hierarchy, forcing the parent `bubble` Box to request maximum width. This neutralized `halign(Align::End)`, as the bubble was already spanning 100% of the available layout.
**Resolution:** Explicitly declared `bubble.set_hexpand(false)` upon instantiation and stripped `.hexpand(true)` from `meta_label`. The `bubble` now strictly conforms to the natural width of its content (clamped by `msg_label`'s `max_width_chars`), allowing `halign` to correctly physically move the bounding box to the right edge of the `root` container.

## 2026-03-28 - Repairing Header Crush and Stagger on Ultra-Wide

## 2026-03-28 - Purging Hardcoded Asymmetric Margins for Responsive GTK4 Text Wrapping
**Anomaly:** Injecting massive 128px hardcoded margins created a static hack that successfully staggered chat bubbles on a 2880px wide display, but fundamentally destroyed responsive text wrapping when the window was compressed.
**Resolution:** Restored standard `max_width_chars(85)` to `msg_label` to let Pango dictate natural expansion boundaries. Replaced the 128px hack with sensible 64px flank protection margins and 12px edge padding. By enforcing `root.set_halign(gtk4::Align::Fill)` on the outer row container and `bubble.set_hexpand(false)` on the inner box, GTK perfectly delegates horizontal positioning to `halign(Align::End)`, achieving a dramatic, completely fluid right-alignment stagger on ultra-wide screens without breaking narrow form factors.

## 2026-03-28 - Designing the Bulletproof GTK4 Chat Stagger via Hexing Spacers
**Anomaly:** Relying on `halign(Align::Fill)` on the root row and explicit padded margins caused inconsistent reflowing behavior across mixed resolutions. When compressed, the bubble text jumped, and when expanded to 2880px, the engine struggled to balance the maximum character constraints against the rigid 64px flank protections.
**Resolution:** Replaced the margin/alignment hack with structural expanding "Spacers". The `root` `ListItem` container is now a horizontal `Box` containing `[left_spacer, bubble, right_spacer]`. The spacers natively `.set_hexpand(true)`, aggressively absorbing 100% of the available empty layout space. Toggling `left_spacer.set_visible(true)` physically shoves the rigid, text-wrapped bubble to the exact right edge of the screen, creating a bulletproof, fluid right-alignment stagger that never tears text or touches the monitor bezel.

## 2026-03-28 - Killing GTK4 hexpand Propagation Infection from Inner Labels
**Anomaly:** Even with outer spacers correctly configured to absorb wide-screen variance, the `bubble` still aggressively stretched to 100% width on a 2880px screen. This occurred because a deeply nested `meta_label` was set to `.hexpand(true)`. In GTK4, if any child widget deep inside a layout tree requests horizontal expansion, that property aggressively bubbles up the entire hierarchy, completely overriding `.set_hexpand(false)` on its parent containers.
**Resolution:** Surgically extracted `.hexpand(true)` from the `meta_label` to stop the infection, replacing it with `.hexpand(false).halign(gtk4::Align::Fill)`. To ensure the adjacent `right_expand_btn` still rests cleanly on the right edge of the header, it was configured with `.halign(gtk4::Align::End).hexpand(true)`. This snaps the button precisely to the rightmost boundary permitted by the text wrapping engine, without artificially stretching the `bubble` container itself.

## 2026-03-28 - Restoring Pure Vertical Float Architecture via Hexing Decoupling
**Anomaly:** Structural expanding "Spacers" created a rigid layout conflict against natural Pango text wrapping. A spacer aggressively absorbing 100% of the horizontal variance forces the bubble to abruptly snap to the edge rather than smoothly scaling its inner width up to its character limit.
**Resolution:** Abandoned the horizontal spacer hack in favor of a pure vertical layout float. By fully decoupling horizontal expansion properties (`root.set_hexpand(true).halign(Align::Fill)` alongside `bubble.set_hexpand(false)` and `meta_label.set_hexpand(false)`), the bubble's width is dictated entirely by `msg_label.max_width_chars(85)`. This allows the bubble to smoothly expand its text bounds while floating natively to the `halign(Align::End)` or `halign(Align::Start)` boundary of the 2880px `root` container, buffered cleanly by soft 64px flank margins.
