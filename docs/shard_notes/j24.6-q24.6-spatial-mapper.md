## 2024-10-24 - The Spatial Cartographer: Resolving the IPC vs. LLM Context Tension

**Anomaly:** The `DICTIONARY$TOPOLOGY` compressed format used for the OS Matrix DAG, while brilliant for passing low-latency data across `async_channel` thread boundaries to the GTK/Qt UI layer, presents a high cognitive load for the LLM when included in the PreFlightPayload. The attention mechanism of the LLM wastes tokens acting as a pointer-dereferencer (`0:1,2 | 1:3`), degrading reasoning quality and increasing the risk of hallucination when trying to understand spatial code topologies. Furthermore, the `MatrixScanner::scan_file` lexical extraction engine was too rudimentary to capture complex Rust dependency mappings (like visibility modifiers and nested `{}` imports).

**Resolution:**
1. The `MatrixScanner::scan_file` logic was completely rewritten using strict Can-Am rules. It now employs pure, zero-allocation string slice tokenization (`&str`) to parse `mod` and `use` dependencies, explicitly expanding multi-line paths and nested `{}` paths using a stack without needing an external AST parser.
2. The core architectural revelation is to maintain the DICTIONARY$TOPOLOGY format for the UI to preserve IPC/memory efficiency. However, just before the `PreFlightPayload` is constructed, this data must be decompressed into a semantic format (like an indented tree or explicit node->dependency list) before it reaches the AI context window. This separates machine code formatting (UI) from semantic context formatting (LLM).
