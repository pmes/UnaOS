## 2026-05-18 - [Architectural Audit & Blueprint: Phase 2 Execution]

**Anomaly:** `textView:doCommandBySelector:` using `*const c_void` causing runtime panics at startup.
**Resolution:** The macro strictly validates Objective-C type encodings. A raw `c_void` pointer evaluates as `^v` whereas a selector evaluates as `:`. Used `objc2::sel::Sel` for the `command_selector` argument to satisfy the macro's strict typing and prevent application crashes during class registration.

**Anomaly:** Unbounded memory leak when calling `dispatch_async` using `block2::RcBlock`.
**Resolution:** Calling `std::mem::forget(block)` on the Rust side after passing it to GCD caused an unbounded leak for every telemetry tick. Because `dispatch_async` inherently copies the block (`Block_copy`) and retains it for execution, the Rust-side `RcBlock` MUST be dropped locally when it goes out of scope to maintain the proper retain count.

**Anomaly:** Potential dangling pointers or memory leaks for root UI delegates.
**Resolution:** Instead of calling `std::mem::forget` on root tab controllers and data source/delegates (which prevents dangling pointers but is bad architecture), we introduced `WorkspaceRefs`, `SidebarRefs`, and `CommsRefs` structs. These structs hold strong `Retained` references to the UI hierarchy and are stored in the `AppDelegateIvars`'s `RefCell`, tying their lifecycle cleanly to the `NSApplicationDelegate`.

**Anomaly:** Compiler rejection: `no rules expected 'DeclaredClass'` matching `DefinedClass` inside `declare_class!`.
**Resolution:** The initial architecture relied on deprecated `objc2` `v0.4.x` macro syntax while standardizing on version `0.6.4`. The macro was completely overhauled in `v0.5+`. All instances of `declare_class!` must be strictly migrated to `define_class!`, and `impl DeclaredClass` blocks converted to `impl DefinedClass`.
